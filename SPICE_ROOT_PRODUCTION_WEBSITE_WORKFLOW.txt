SPICE ROOT PRODUCTION WEBSITE WORKFLOW DOCUMENTATION
====================================================

Document purpose
----------------
This document explains the complete production-level workflow for the Spice Root website from scratch.
It covers planning, architecture, pages, user flows, admin flows, Firebase data, contact reply notifications,
Razorpay test payments, security, testing, deployment, launch, and maintenance.

Project name: Spice Root
Website type: Customer e-commerce/catalog website with admin dashboard
Frontend stack: React, Vite, React Router, Firebase, Firestore
Styling: Custom CSS modules/files
Authentication: Firebase Authentication
Database: Cloud Firestore
Storage: Firebase Storage
Payment gateway: Razorpay test checkout
Admin panel: Custom React admin dashboard


1. PROJECT GOALS
----------------
The website must allow customers to:

1. Browse Spice Root products.
2. View product details.
3. Register and log in.
4. Add products to cart.
5. Checkout with delivery details.
6. Complete Razorpay test payment before order placement.
7. View order history.
8. Contact the admin/team using Contact Us.
9. Receive notification when admin replies to a contact query.
10. Read admin replies and clear notification state.
11. Submit product/store reviews after delivered orders.

The admin panel must allow admins to:

1. Log in through the admin route.
2. View dashboard KPIs.
3. Manage products.
4. Manage categories.
5. View orders.
6. Update order status.
7. View payments and revenue.
8. View customers.
9. View customer details.
10. View and reply to contact queries.
11. Manage reviews.
12. Track inventory.
13. View notification panels for new orders and new queries.

3. MAIN TECHNOLOGY WORKFLOW
---------------------------
Frontend:

1. React renders all customer and admin UI.
2. React Router controls website navigation.
3. Vite builds and serves the app.
4. Firebase Auth handles login/register.
5. Firestore stores users, carts, orders, products, reviews, contact queries, notifications, and payments.
6. Firebase Storage is used for product media when needed.
7. Razorpay test checkout opens from the checkout page.

Backend:

This app currently uses Firebase client SDK and Firestore security rules instead of a custom backend.
For production-grade real Razorpay verification, add a backend or Firebase Cloud Functions because payment
signature verification requires a secret key that must never run in browser code.

5.5 Checkout page

Purpose:
Users enter delivery details, complete Razorpay test payment, and only then place the order.

Required fields:

1. Full name
2. Mobile number
3. Alternate mobile number, optional
4. Address line 1
5. Address line 2, optional
6. City
7. State
8. Pincode
9. Delivery instructions, optional

Validation:

1. Full name is required.
2. Mobile number must be 10 digits.
3. Alternate mobile number, when provided, must be 10 digits.
4. Address line 1 is required.
5. City is required.
6. State is required.
7. Pincode must be 6 digits.
8. Cart must not be empty.
9. User must be authenticated.

Payment flow:

1. User fills valid delivery form.
2. App saves customer profile details when possible.
3. App prepares order payload in memory.
4. App opens Razorpay test checkout.
5. If user closes payment modal, no order is created.
6. If test payment fails, no order is created.
7. If test payment succeeds, order is written to Firestore.
8. Cart is cleared.
9. Success view displays the newly placed order.
10. Order appears in user order history.
11. Admin can see order.
12. Admin can see payment record and revenue.

Important production note:
Current Razorpay integration is test/demo mode only. A real production Razorpay integration must use a server-side
order creation endpoint and server-side signature verification. The Razorpay secret key must never be exposed in client JavaScript.

6. USER NOTIFICATION WORKFLOW
-----------------------------

Problem solved:
The user was not receiving visible notification from admin replies to Contact Us queries.

Correct source of truth:
The notification state is now based on contactSubmissions:

adminReply exists
userReplySeen is false

Notification display rule:

Show bell when:

1. User is logged in.
2. At least one contact submission belongs to user by userId or email.
3. Submission has adminReply.
4. userReplySeen is false.

Notification clear rule:

When user opens notification popup:

1. Popup shows admin reply.
2. App updates contactSubmissions/{submissionId}.
3. userReplySeen becomes true.
4. Bell count disappears after Firestore sync.

Important UI fix:
The notification popup uses a React portal to render at document body level. This prevents it from being clipped or hidden by the navbar/header stacking context.

11. RAZORPAY PAYMENT WORKFLOW
-----------------------------

Current mode:
Test/demo frontend checkout only.

Current payment behavior:

1. User validates checkout form.
2. Razorpay checkout script loads.
3. Test checkout opens.
4. On success, app receives payment id.
5. App creates Firestore order.
6. App sets paymentStatus to Paid.
7. Admin Payments shows payment record.
8. Admin Dashboard revenue counts paid order.

Production-ready Razorpay workflow:

1. User clicks Pay.
2. Frontend sends cart/order summary to backend.
3. Backend validates price and cart.
4. Backend creates Razorpay order using key secret.
5. Frontend opens Razorpay checkout with backend order id.
6. Razorpay returns payment id, order id, signature.
7. Frontend sends result to backend.
8. Backend verifies signature using Razorpay secret.
9. Backend creates order in Firestore only after verification.
10. Backend creates payment record.
11. Backend returns success to frontend.
12. Frontend clears cart and shows success.

Why server verification is required:
Browser code can be modified by users. A browser-only payment success callback is acceptable for a demo/test mode,
but not enough for real money orders.


12. SECURITY WORKFLOW
---------------------

Authentication:

1. Use Firebase Auth.
2. Protect customer routes.
3. Protect admin routes.
4. Prevent admin users from customer checkout.
5. Use admin allowlist and/or admins collection.

Authorization:

1. Enforce ownership in Firestore rules.
2. Enforce admin-only writes for admin operations.
3. Validate allowed fields in rules.
4. Avoid broad allow read/write.

Secrets:

1. Do not expose Razorpay secret.
2. Do not expose service account credentials.
3. Do not commit .env.local.
4. Use deployment secrets.

Payments:

1. Keep current Razorpay mode as test only.
2. Add backend verification before real payments.
3. Do not trust client totals.
4. Backend should recalculate totals from product data.

Data validation:

1. Validate form fields before writes.
2. Normalize statuses.
3. Use serverTimestamp for important write times.
4. Restrict admin updates to known fields.
